# Runtime 迁移治理规范

> 版本：v1.0（草案）　|　日期：2026-06-19　|　编制：韩衡（质量管理团队）　|　裁决：何明（质量管理 Leader）/ CTO　|　状态：待何明定稿
>
> 来源：SUP-347 demo 日 runtime 双切换致编码产能空窗复盘；何明评审结论：5 条改进建议原则采纳，其中 4 条纳入 SUP-99 方法论库。

本规范用于治理 SmartDesk 项目中 AI agent 的 runtime 迁移行为，防止迁移操作在冲刺/demo 窗口造成产能空窗，并确保迁移前后有完整的检查、通知与记录闭环。

---

## 1. 冲刺/demo 窗口冻结 runtime 迁移

### 1.1 冻结范围

| 场景 | 冻结时间 | 说明 |
|---|---|---|
| demo 冲刺日 | demo 冲刺当日 00:00（CST）至 demo 演示结束 | 以 PMO 发布的 demo 冲刺日历为准 |
| 重大里程碑冲刺 | 里程碑截止前 24h 至里程碑验收完成 | 由 PMO 在里程碑计划中明确 |
| 发布窗口 | 灰度/全量发布前 12h 至发布观察期结束 | 与 `quality/review-checklist.md`、发布流程对齐 |

### 1.2 冻结对象

- 任何导致编码 agent 批量切换 provider/model/runtime 的 `multica agent update` 操作；
- 任何改变 agent 运行时环境（模型、thinking level、runtime mode 等）并可能导致其短期不可用或行为漂移的变更；
- 单次影响 3 个及以上编码 agent 的迁移视为「批量迁移」，适用本条款。

### 1.3 例外审批路径

| 情形 | 审批人 | 审批动作 | 记录位置 |
|---|---|---|---|
| 常规冻结期内迁移 | 禁止 | — | 违者入红黑台账 |
| 紧急迁移（如 provider 额度耗尽影响交付）| PMO Leader 程远初审 → CTO 终审 | 在对应 issue 评论中书面批准 | SUP-99 方法论库事件索引 + 红黑台账 |
| 极端紧急且 CTO 不可用时 | CEO（Allen）直接裁决 | 事后 30min 内补录书面说明 | 红黑台账 |

### 1.4 紧急迁移的降级方案

若冻结期内确需迁移，必须同步执行以下降级，由 PMO 在审批评论中明确：

1. **分批迁移**：一次最多迁移 2 个 agent，确保其余 agent 维持产能；
2. **热备就绪**：目标 runtime 已提前验证可运行项目任务（非首次使用）；
3. **PMO 现场值守**：迁移期间 PMO Leader 或协调员在线，迁移完成后立即执行 §2「5 分钟派活检查」；
4. **回滚预案**：原 runtime 配置已备份，可在 10 分钟内回滚；
5. **通知到位**：在相关 issue/群聊中 @ 所有受影响 agent 的负责人。

## 2. 迁移完成后 5 分钟内派活检查

### 2.1 触发条件

以下任一情形发生后，立即触发本检查：

- 任何 agent 完成 runtime 迁移（无论是否批量）；
- 任何 agent 因 runtime 问题被回滚或重启；
- 任何导致 agent 从 idle/不可用到可用状态恢复的运维操作。

### 2.2 责任人

| 环节 | 责任人 | 说明 |
|---|---|---|
| 迁移执行人 | 发起迁移的 agent/工程师 | 负责在迁移完成后 1 分钟内通知 PMO |
| 派活检查 | PMO Leader 程远或协调员方通 | 负责在 5 分钟内确认 agent 可用并派出任务 |
| 备用检查 | 相关团队 Leader | 若 PMO 不可用，由受影响 agent 所属团队 Leader 兜底 |

### 2.3 检查动作

**人工检查（当前最小可行）**：

1. 登录 Multica 看板，确认受影响 agent 状态为 `idle` 或 `working`；
2. 向每个受影响 agent 派出一条**非阻塞验证任务**（如：检查当前环境、读取并回复一个简单问题）；
3. 观察 agent 是否在 2 分钟内开始执行；
4. 若 agent 未响应，立即尝试重新唤醒或回滚。

**Autopilot 检查（目标态）**：

- 由 CTO/平台团队评估建立 autopilot：监听 `agent.runtime.updated` 事件，在迁移完成后自动向受影响 agent 派发健康检查任务；
- 若健康检查 2 分钟未通过，自动通知 PMO 并建议回滚；
- 该 autopilot 须经 CTO 批准后方可启用，避免误触发。

### 2.4 升级路径

| 检查结果 | 动作 | 升级时限 |
|---|---|---|
| 5 分钟内成功派活并响应 | 记录检查日志，关闭本节点 | — |
| 5 分钟内未派活成功 | PMO 立即在 issue 评论中说明原因并 @ 相关 Leader | 立即 |
| 10 分钟内仍无法恢复产能 | 升级 PMO Leader 程远 → CTO | T+10min |
| 影响 demo/里程碑交付 | 直接升级 CEO（Allen） | 立即 |

### 2.5 记录模板

迁移完成后，迁移执行人须在对应 issue 中评论：

```markdown
## Runtime 迁移完成记录

- 迁移时间（UTC）：
- 受影响 agent：
- 迁移类型：codex→opencode / opencode→codex / 其它
- PMO 通知时间：
- 派活检查完成时间：
- 检查结果：通过 / 未通过
- 未通过原因及升级对象：
```

## 3. 迁移前 Checklist

### 3.1 强制检查项

| 类别 | 检查项 | 通过标准 | 责任人 |
|---|---|---|---|
| 配置 | 目标 runtime 已配置项目所需模型与参数 | `multica agent get <id>` 确认 model/thinking_level/runtime_mode 符合预期 | 迁移执行人 |
| 配置 | 原 runtime 配置已备份 | 在 issue/评论中记录原 model、runtime_mode、thinking_level | 迁移执行人 |
| 依赖 | 目标 runtime 支持项目必要工具/MCP | 已验证可调用 git、代码编辑、测试等工具 | 迁移执行人 |
| 依赖 | 目标 runtime 不缺失关键 CLI 参数 | 例如 `--thinking-level` 已在目标 runtime 可用 | 迁移执行人 |
| 回滚 | 回滚方案已就绪 | 可在 10 分钟内切回原配置并验证 | 迁移执行人 |
| 通知 | 受影响 agent 负责人已知情 | 在 issue 中 @ 相关 Leader/agent | 迁移执行人 |
| 窗口 | 当前不在冻结窗口内 | 对照 §1.1 冻结范围确认；若属例外，已获 CTO/CEO 书面批准 | PMO |
| 值守 | PMO 或团队 Leader 可在迁移后 15 分钟内在线 | 确认有人执行 §2 派活检查 | PMO |

### 3.2 模板

迁移执行人在发起迁移前，须在对应 issue 评论中复制并勾选以下 checklist：

```markdown
### Runtime 迁移前 Checklist

- [ ] 目标 runtime 模型与参数已确认符合项目需求
- [ ] 原 runtime 配置已备份（model / runtime_mode / thinking_level）
- [ ] 目标 runtime 已验证支持项目必要工具/MCP
- [ ] 目标 runtime 不缺失关键 CLI 参数（如 `--thinking-level`）
- [ ] 回滚方案已就绪，可在 10 分钟内回滚并验证
- [ ] 受影响 agent 负责人已在 issue 中被 @ 知情
- [ ] 当前不在冲刺/demo/发布冻结窗口，或已获 CTO/CEO 书面例外批准
- [ ] PMO/团队 Leader 可在迁移后 15 分钟内在线执行派活检查
```

### 3.3 未通过处理

任一强制检查项未通过，禁止执行迁移。违者按 §4 入红黑事件台账。

## 4. 红黑事件台账治理

### 4.1 台账定位

- 维护方：质量管理团队（韩衡）；
- 原则：**对事不对人、无责记录**；
- 用途：记录研发流程中的亮点（红）与问题/漂移（黑），用于趋势分析与改进闭环。

### 4.2 字段定义

| 字段 | 说明 |
|---|---|
| 编号 | RB-YYYYMMDD-NNN（日期 + 当日序号） |
| 类型 | 红 / 黑 |
| 来源 | 触发事件的 issue、巡检、评审或复盘 |
| 事件 | 一句话概括 |
| 影响范围 | 涉及的服务/模块/团队/agent |
| 根因 | 简要根因分析 |
| 改进项 | 已创建或待创建的改进 issue |
| 状态 | 跟踪中 / 已关闭 / 待验证 |
| 记录人 | 记录者 |

### 4.3 录入责任人

| 事件来源 | 录入责任人 | 时限 |
|---|---|---|
| 复盘 issue（如 SUP-347） | 韩衡 | 复盘结论产出后 30 分钟内 |
| 质量巡检（睿恒/韩衡） | 韩衡 | 巡检发现后 30 分钟内 |
| 评审会议/CTO 裁决 | 韩衡 | 裁决产出后 30 分钟内 |
| 各团队主动上报 | 上报人发起，韩衡补录/审核 | 上报后 1h 内 |

### 4.4 更新节奏

- **新增事件**：事件确认后即时录入；
- **状态更新**：事件关闭或验证通过后 30 分钟内更新状态；
- **趋势汇总**：每周由韩衡输出一次红黑事件趋势简报，按模块/团队统计缺陷类型，识别反复出现的基本功能/流程问题并量化，提请何明要求相关 Leader 改进。

### 4.5 主文档

红黑事件台账主文档见 `quality/red-black-ledger.md`。
