## 1. 目标与范围

为 SUP-189（四仓 CI/CD 金丝雀 deploy/rollback job）补充**流水线级监控告警**，确保 G1–G5 门禁期间任何异常都能及时通知到值班、发布经理（白帆）与 CTO（Allen Li）。

**覆盖场景**：

1. `deploy-canary.yml` / `rollback.yml` 任一步骤失败时通知值班与 CTO。
2. GitHub Environment required reviewer 长时间未审批时提醒。
3. C1/C2/C3 与 rollback 成功后通知发布群，并提示下一门禁待确认。
4. 镜像构建失败、GHCR 推送失败、compose 脚本退出码非零时告警。

**不在本稿范围**：Prometheus/Alertmanager 服务指标告警规则（已由 [`发布监控告警基线.md`](发布监控告警基线.md) 定义，待 staging 就绪后落地）。

## 2. 方案总览

采用「**复用 Action + 流水线调用**」的轻量模式，所有通知收敛到同一个 `notify-alert` Action，便于统一 payload 格式与路由。

```
┌─────────────────────────────────────────────────────────────┐
│  GitHub Actions                                               │
│  ┌─────────────────┐   ┌─────────────────┐                   │
│  │ deploy-canary   │   │ rollback        │                   │
│  │ .yml            │   │ .yml            │                   │
│  └────────┬────────┘   └────────┬────────┘                   │
│           │                     │                            │
│           ▼                     ▼                            │
│  ┌─────────────────────────────────────┐                    │
│  │  notify-alert (composite action)    │                    │
│  │  • Webhook (Slack/Teams/企业微信)   │                    │
│  │  • SMTP 邮件兜底（可选）            │                    │
│  │  • payload artifact                 │                    │
│  └─────────────────────────────────────┘                    │
└─────────────────────────────────────────────────────────────┘
```

## 3. 复用告警 Action

路径：`.github/actions/notify-alert/action.yml`

核心能力：

| 输入 | 说明 |
|---|---|
| `severity` | `P0` / `P1` / `P2` / `info` / `success` |
| `title` | 告警标题 |
| `body` | 多行正文 |
| `recipients` | 通知对象，逗号分隔（频道/邮箱/用户名） |
| `webhook_url` | 通用 Webhook URL，推荐配置在 `NOTIFY_WEBHOOK_URL` Secret |
| `smtp_*` | 可选 SMTP 兜底 |

输出：

| 输出 | 说明 |
|---|---|
| `sent` | 是否成功 POST 到 Webhook |
| `response` | Webhook 返回内容 |

未配置 Webhook 时，Action **不会失败**，仅记录日志并上传 payload artifact，方便本地调试与后续审计。

## 4. 灰度发布流水线告警

路径：`.github/workflows/deploy-canary.yml`

### 4.1 触发方式

- `workflow_dispatch`：手动选择 `stage`（c1/c2/c3）、镜像 tag、ingress 端口等。
- 未来可由四服务仓库通过 `workflow_call` 或 `repository_dispatch` 调用（本文档版本以手动触发与模板为主）。

### 4.2 门禁人类确认

每个 stage 均通过 GitHub Environment 实现 required reviewers：

| Environment | 对应门禁 | 说明 |
|---|---|---|
| `canary-gate-c1` | G2 | CTO 确认 C1（5%）启动 |
| `canary-gate-c2` | G3 | CTO 确认 C1 → C2 推进 |
| `canary-gate-c3` | G4 | CTO 确认 C2 → C3 推进 |

配置要求（仓库管理员操作）：

1. Settings → Environments → New environment。
2. 名称分别为 `canary-gate-c1`、`canary-gate-c2`、`canary-gate-c3`。
3. Deployment protection rules → Required reviewers 添加 CTO（`multica-test@hotmail.com`）。

每个 gate job 设置 `timeout-minutes: 60`，超时未审批则 job 失败，触发 **P1 超时告警**。

### 4.3 告警映射

| 场景 | 触发位置 | 级别 | 通知对象 |
|---|---|---|---|
| 流水线启动 | `notify-start` | info | 发布群 |
| 镜像构建/推送失败 | `build-canary-stub` 失败 → `notify-failure` | P0 | 值班 + CTO + 白帆 |
| G 门禁人工确认超时 | `gate` 超时 → `notify-gate-timeout` | P1 | 白帆 + CTO |
| compose 部署脚本非零退出 | `deploy-stage` 失败 → `notify-failure` | P0 | 值班 + CTO + 白帆 |
| 分流比例验证失败 | `deploy-stage` 失败 → `notify-failure` | P0 | 值班 + CTO + 白帆 |
| 阶段部署成功 | `promote` | success | 发布群；提示下一门禁 |
| 任意 job 失败兜底 | `notify-failure` | P0 | 值班 + CTO + 白帆 |

> 与 [`发布监控告警基线.md`](发布监控告警基线.md) 对齐：流水线失败视为**发布过程异常**，统一按 P0 处理；仅门禁超时按 P1 处理。

## 5. 回滚流水线告警

路径：`.github/workflows/rollback.yml`

| 场景 | 触发位置 | 级别 | 通知对象 |
|---|---|---|---|
| 回滚启动 | `notify-start` | P0 | 值班 + CTO + 白帆（回滚本身就是重大事件） |
| compose 回滚脚本失败 | `rollback` 失败 → `notify-failure` | P0 | 值班 + CTO + 白帆 |
| 回滚后健康检查/冒烟失败 | `verify` 失败 → `notify-failure` | P0 | 石磊/苏睿/关山 + 白帆 |
| 回滚成功 | `notify-success` | success | 发布群；提示继续观测 15 min |
| 任意 job 失败兜底 | `notify-failure` | P0 | 值班 + CTO + 白帆 |

回滚顺序与 [`回滚预案.md`](回滚预案.md) §4 一致：`web → gateway → insight → core`。

## 6. 与监控基线的对齐

本配置解决**流水线级**告警；服务级 P0/P1 告警仍由 Prometheus/Alertmanager 实现，两者互补：

| 维度 | 本流水线配置 | Prometheus 告警规则 |
|---|---|---|
| 失败来源 | GitHub Actions job/step | 服务运行时指标 |
| 响应动作 | 通知人 → 人工裁决/执行 Runbook | 通知人 → 触发自动/人工响应 |
| 触发条件 | 构建失败、部署失败、门禁超时 | 5xx 率、P95、SLI 等阈值 |
| 关联文档 | 本文档 | [`发布监控告警基线.md`](发布监控告警基线.md) |

## 7. 仓库 Secrets 与环境配置

### 7.1 Secrets

| Secret | 用途 | 必填 |
|---|---|---|
| `NOTIFY_WEBHOOK_URL` | 通用告警 Webhook（Slack/Teams/企业微信） | 推荐 |
| `SMTP_SERVER` | SMTP 服务器地址 | 可选 |
| `SMTP_PORT` | SMTP 端口 | 可选 |
| `SMTP_USER` | SMTP 用户名 | 可选 |
| `SMTP_PASSWORD` | SMTP 密码 | 可选 |
| `SMTP_FROM` / `SMTP_TO` | 邮件发件人/收件人 | 可选 |

### 7.2 GitHub Environments

需在仓库设置中创建以下 Environment 并配置 required reviewers：

- `canary-gate-c1`
- `canary-gate-c2`
- `canary-gate-c3`

每个 environment 的 required reviewers 建议为 CTO（`multica-test@hotmail.com`），与 [`灰度发布策略.md`](灰度发布策略.md) §7 G1–G5 人类确认角色一致。
