## User Scenarios & Testing *(mandatory)*

> 系统级用户故事来源于 PRD F1–F7 与用户故事文档 US-1.1~US-7.3。此处按**优先级**归并为可独立交付/测试的故事组；逐条 Given/When/Then 见 [用户故事与验收标准](../SmartDesk用户故事与验收标准.md)。

### User Story 1 - 提单到关闭闭环（Priority: P1 · MVP 核心）
作为**报单人**，我想要一键结构化提单并跟踪其全生命周期到关闭，以便规范、可追溯地获得服务。
**Acceptance Scenarios**:
1. **Given** 标题+描述齐全，**When** 提交，**Then** 建单成功返回工单号、状态=`新建`，且**即使 AI 不可用也成功**（降级）。
2. **Given** 工单处理中，**When** 坐席按状态机推进（受理→处理中→待用户→已解决），**Then** 非法跃迁被拒、每次变更写时间线、重复提交幂等。
3. **Given** 工单`已解决`，**When** 报单人确认，**Then** 转`已关闭`；7 天内可重开，超期需新建。

### User Story 2 - 网关统一鉴权与 RBAC（Priority: P1 · 安全红线）
作为**系统**，我想要在网关收口认证与 RBAC，以便最小权限保护后端。
**Acceptance Scenarios**:
1. **Given** 未经鉴权的直达后端请求，**When** 到达 core/insight，**Then** 被拒（仅信任 gateway）。
2. **Given** `requester`，**When** 访问他人工单/管理后台/内部备注，**Then** 403 且记审计。
3. **Given** 超速率阈值，**When** 持续请求，**Then** 429。

### User Story 3 - AI 辅助分析（异步、不阻塞）（Priority: P2）
作为**坐席**，我想要系统在建单后异步给出分类/定级/相似建议，以便减少人工并复用经验。
**Acceptance Scenarios**:
1. **Given** 建单完成，**When** `ticket.created` 事件被 insight 消费，**Then** 异步算出建议并写回 core 为"建议态"（一期不自动填充）。
2. **Given** AI/总线故障，**When** 打开工单，**Then** 建议为空/相似懒加载降级，主流程不受影响。
3. **Given** 坐席纠偏分类，**When** 保存，**Then** 纠偏样本回流 insight 供迭代。

### User Story 4 - 协作、通知与事件（Priority: P2）
作为**用户**，我想要转派/升级/@提及/关注，并在站内+邮件收到与我相关的更新，以便及时协作。
**Acceptance Scenarios**:
1. **Given** 关键业务动作（建单/分派/状态/评论/SLA 预警/解决/关闭），**When** 发生，**Then** 发布统一 schema 领域事件，至少一次送达、消费幂等。
2. **Given** @提及无权限查看该工单者，**When** 评论，**Then** 不发送通知（鉴权校验）。

### User Story 5 - SLA 计时与预警（Priority: P2）
作为**系统/组长**，我想要按优先级计 SLA 并临近/超时预警，以便守住时限。
**Acceptance Scenarios**:
1. **Given** 优先级，**When** 建单/受理，**Then** 启动响应+解决计时。
2. **Given** `待用户`，**When** 进入/退出，**Then** 计时暂停/恢复顺延。
3. **Given** 临近阈值/超时，**When** 触发，**Then** 发 `sla_warning`/`sla_breached` 事件、通知坐席+组长、超时标记。

### User Story 6 - 看板与质量报表（Priority: P3）
作为**管理者/组长**，我想要近实时看板（SLA 达成/积压/工作量/分类分布/重开率/CSAT）并可导出，以便掌握全局质量。
**Acceptance Scenarios**:
1. **Given** 领域事件流，**When** insight 投影统计，**Then** 看板近实时、聚合与明细同口径。
2. **Given** 当前筛选，**When** 导出，**Then** 生成 CSV。

### User Story 7 - 三端前端（Priority: P1–P3 随后端能力推进）
作为**报单人/坐席/管理员**，我想要门户/工作台/后台三端 UI，以便低门槛使用与运营。
**Acceptance Scenarios**:
1. 报单门户：一键提单 + 我的工单（确认/重开）。
2. 坐席工作台：左队列右详情（时间线/评论备注/相似/状态操作）。
3. 管理后台：分类树/SLA 策略/用户角色/通知策略 CRUD。

### Edge Cases
- AI/事件总线宕机 → 建单与状态流转不受影响；建议为空。
- 状态非法跃迁、`待用户`超 7 天 → 拒绝跃迁 / 仅提醒不自动关闭。
- 合并工单 → 子单 `merged_into` 主单、状态同步、通知报单人。
- 附件超 20MB/非白名单类型 → 413/拒绝；越权下载 → 403。
- 内部备注对 requester → 接口层过滤，不仅 UI。
- 事件重复投递 → 按 `event_id` 去重，无重复通知/写回。

## Requirements *(mandatory)*

### Functional Requirements（系统级，映射 PRD F1–F7）
- **FR-001（F1）**: 系统 MUST 在 gateway 收口认证（登录/刷新/登出/me、JWT、会话）与 RBAC（角色×动作矩阵，未授权 403 且记审计），并提供统一聚合入口与限流（429）。
- **FR-002（F2）**: 系统 MUST 支持工单全生命周期：结构化建单、八态状态机（非法跃迁拒绝、幂等、写时间线）、分派/改派/自动分派、对外评论与内部备注（接口层可见性过滤）、附件（≤20MB/白名单/对象存储/下载鉴权）、关联/合并、SLA 计时（暂停/恢复/预警/超时）、时间线/审计（仅追加）。
- **FR-003（F3）**: 系统 MUST 在建单后**异步**产出分类/定级/相似建议并写回 core 为建议态（一期不自动填充，阈值 ≥0.85 可配）；检索一期关键词+标题/分类、契约前向兼容向量；纠偏样本回流。
- **FR-004（F4）**: 系统 MUST 发布统一 schema 领域事件（至少一次、消费幂等），提供站内+邮件通知与 admin 通知策略。
- **FR-005（F5）**: 系统 MUST 支持转派/升级、@提及（含权限校验）、关注/跟随。
- **FR-006（F6）**: 系统 MUST 提供近实时看板与质量视图（SLA 达成/积压/工作量/分类分布/重开率/CSAT）与 CSV 导出。
- **FR-007（F7）**: 系统 MUST 提供报单门户/坐席工作台/管理后台三端 UI，仅经 gateway 调用。
- **FR-008（降级）**: 核心建单链路 MUST 在 AI/事件总线不可用时仍成功（FR-003 异步不阻塞 FR-002）。
- **FR-009（多租户预留）**: 所有业务数据 MUST 携带 `org_id`，一期单组织、不实现隔离逻辑。
- **FR-010（合规）**: 系统 MUST 提供 PII 最小化、审计不可篡改、附件鉴权、admin 导出/删除；留存期/法务在 M4 GA 前闭环。

### Key Entities
- **Ticket（工单）**: 服务请求载体；八态、优先级 P1–P4、流水号、重开计数、CSAT；归属 core。
- **User / Role（账号与角色）**: RBAC 主数据归 core；认证凭证归 gateway。
- **Category（分类树）**: admin 可维护，归 core 权威。
- **SLA Policy / Timer（策略与计时）**: 优先级→时限，可暂停；归 core。
- **Comment / Attachment / Link / Timeline（评论/附件/关联/时间线）**: 归 core。
- **Domain Event（领域事件）**: 统一信封；core 发布、insight 消费。
- **Suggestion / Similarity / Notification / Stats（建议/相似/通知/统计）**: 归 insight 读模型。

## Success Criteria *(mandatory)*

### Measurable Outcomes
- **SC-001**: 报单人可在 3 分钟内完成一次结构化提单到提交。
- **SC-002**: 建单链路在 AI 不可用时成功率 100%（降级不影响建单）。
- **SC-003**: 工单列表/详情 P95 < 500ms（常规负载）。
- **SC-004**: AI 分类/相似建议 P95 < 2s 且不阻塞建单主流程。
- **SC-005**: SLA 达成率达到 ≥95% 目标（v1 基线，可配阈值）。
- **SC-006**: 越权/鉴权用例 100% 拦截（403），无未经 gateway 的后端直达。
- **SC-007**: 领域事件至少一次送达、重复投递 0 重复副作用（按 `event_id` 幂等）。
- **SC-008**: 系统目标可用性 ≥99.5%。

## Assumptions
- 一期单组织（`org_id=default`），不实现租户隔离逻辑（OQ-7）。
- 一期自建账号认证，OIDC 仅预留接口（OQ-2）。
- 一期检索关键词+标题/分类，向量留二期（OQ-5）。
- SLA 数值为 v1 基线、可由 admin 配置（OQ-1）。
- 事件总线选 NATS JetStream（RabbitMQ 等价备选，与事件 schema 解耦）。

## Out of Scope（一期不做）
- 移动端原生 App（仅响应式 Web）。
- 邮件/IM 入站自动建单（OQ-3，二期）。
- 知识库/自助 FAQ（OQ-6，二期）。
- 多租户隔离逻辑（OQ-7，仅预留维度）。
- 向量检索实现（OQ-5，二期）；自动填充默认关闭（OQ-4）。
- 复杂 BI/自助报表设计器（仅预置看板）。
