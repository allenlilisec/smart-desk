## 1. 范围与目标

### 1.1 适用范围

本预案覆盖 SmartDesk **四服务**（gateway / core / insight / web）及共享依赖（PostgreSQL、Redis、NATS JetStream、S3/MinIO）在**灰度发布或全量上线**过程中，因异常需**一键回退到上一稳定版本（Last Known Good, LKG）** 的操作流程。

| 组件 | 仓库 | 运行时 | 健康探针 |
|---|---|---|---|
| **gateway** | `allenlilisec/smartdesk-gateway` | Go | `/healthz`（liveness）、`/readyz`（Redis + 下游可达） |
| **core** | `allenlilisec/smartdesk-core` | Go | `/healthz`（liveness）、`/readyz`（服务可接收流量，不强制探测 DB/NATS） |
| **insight** | `allenlilisec/smartdesk-insight` | Python/FastAPI | `/healthz`、`/readyz`（DB + NATS） |
| **web** | `allenlilisec/smartdesk-web` | Node 20 / Next.js | `/healthz` |
| **PostgreSQL** | — | 共享 | `pg_isready` / 连接池探针（core `readyz` 不探测 DB） |
| **Redis** | — | gateway 会话/限流 | gateway `readyz` 间接探测 |
| **NATS JetStream** | — | 事件总线 `SMARTDESK_EVENTS` | 基础设施探针；insight `readyz` 间接探测（core `readyz` 不探测 NATS） |
| **S3/MinIO** | — | 附件对象存储 | 附件链路冒烟验证 |

### 1.2 时限目标（RTO）

| 阶段 | 目标 | 说明 |
|---|---|---|
| **决策** | ≤ 3 min | 白帆或代行人在观测到触发条件后完成 Go/No-Go（回滚）裁决 |
| **执行** | ≤ 10 min | 完成四服务镜像回退 + 流量切回 LKG |
| **验证** | ≤ 5 min | 核心路径冒烟通过（§5 P0 清单） |
| **通报** | ≤ 2 min | 完成干系人通报（§6） |
| **合计（核心路径）** | **≤ 15 min** | 从触发到核心建单链路恢复可用 |

> **RPO**：数据库采用**向前兼容迁移**策略（§4.3），回滚镜像**不执行 down 迁移**时，RPO ≈ 0（无数据丢失）；若必须 down 迁移，RPO 取决于迁移窗口内写入量，需白帆 + 石磊联合裁决。

### 1.3 前置条件

- 每次发布前由白帆维护 **LKG 版本清单**（四服务镜像 tag + 配置快照 hash + DB schema 版本号）。
- CI/CD 流水线保留 LKG 镜像至少 **3 个发布周期**。
- 灰度发布按 `deploy-canary` 技能执行：**5% → 25% → 100%** 分批放量，每批观测后再推进。
- 本预案 dry-run 验证可在 P3 实现骨架就绪后执行（issue 门禁要求）。

## 2. 回滚触发条件

触发条件与灰度观测阈值对齐。满足**任一**条件即**暂停放量**；满足**任一强制回滚**条件即启动本预案 §4。

### 2.1 暂停放量（Hold）

| # | 指标 / 现象 | 阈值（相对 LKG 基线或 SLO） | 观测窗口 |
|---|---|---|---|
| H-1 | 5xx 错误率 | 较基线上升 **> 0.5%** 或绝对值 **> 1%** | 连续 2 min |
| H-2 | API P95 延迟 | 较基线恶化 **> 30%** 或 gateway 聚合 **> 800ms**（衔接 P1-05 下沿，灰度暂停早警） | 连续 3 min |
| H-3 | `/readyz` 非 200 | 任一服务 readiness 失败 | 连续 2 次探测（间隔 15s） |
| H-4 | 事件消费 lag | **`*_consumer_lag_seconds` > 60s** 或持续上升；NATS consumer 条数 lag > 5000 仅作辅助观测 | 连续 5 min |
| H-5 | 金丝雀探针失败 | 灰度实例健康检查失败 | 即时 |

**动作**：停止下一批放量，万全采集快照（指标截图、日志 trace_id、镜像版本），通报白帆研判。

### 2.2 强制回滚（Rollback）

| # | 指标 / 现象 | 阈值 | 备注 |
|---|---|---|---|
| R-1 | 5xx 错误率 | **> 5%** 或较基线上升 **> 3%** | 影响用户可用性 |
| R-2 | 核心建单失败 | `POST /api/v1/tickets` 成功率 **< 99%**（先于 P0-03 < 95% 触发） | 违反降级红线（US-2.1 AC4） |
| R-3 | 鉴权全面失效 | SLI-02 鉴权失败率 **> 5%**（持续 2 min）；等价 `POST /api/v1/auth/login` 成功率 **< 95%** | 全站不可用；详见 [`灰度发布策略.md`](灰度发布策略.md) §6.2 |
| R-4 | 数据一致性事故 | 状态机非法跃迁批量出现、outbox 堆积不可恢复 | 需石磊确认 |
| R-5 | 安全事件 | 越权访问可复现、service-jwt 绕过 | 武安确认后即时回滚 |
| R-6 | 人工裁决 | 白帆 / CTO / 人类决策者下达回滚指令 | 优先级最高 |

> 灰度阶段若仅金丝雀实例异常且流量可隔离，优先**摘除金丝雀**；全量或摘除非金丝雀后仍不达标，执行完整回滚。

## 3. 版本与配置基线管理

### 3.1 LKG 版本清单（发布前填写）

每次晋升 LKG 前，白帆在发布记录中冻结：

```yaml
lkg:
  recorded_at: "2026-06-15T00:00:00Z"
  release_tag: "v0.2.0-m2"
  images:
    gateway: "registry/smartdesk-gateway:v0.2.0-m2"
    core: "registry/smartdesk-core:v0.2.0-m2"
    insight: "registry/smartdesk-insight:v0.2.0-m2"
    web: "registry/smartdesk-web:v0.2.0-m2"
  config:
    gateway_config_hash: "<sha256>"
    insight_config_hash: "<sha256>"
    web_env_hash: "<sha256>"
  database:
    core_schema_version: "0004"   # golang-migrate 最新 up 版本
    insight_schema_version: "<alembic_rev>"
  dependencies:
    postgresql: "16.x"
    redis: "7.x"
    nats: "2.10.x-jetstream"
```

### 3.2 不可回滚项（需谨慎）

| 项 | 策略 |
|---|---|
| **DB 向前迁移** | 默认**不回滚 schema**（见 §4.3）；新代码依赖新列时，回滚镜像 + 保留 schema |
| **附件对象** | S3/MinIO 对象只追加，回滚不删除 |
| **NATS 事件** | 流保留；消费幂等靠 `event_id` / `processed_events` |
| **已发送通知** | 不可撤回；回滚后靠新业务逻辑兜底 |

## 4. 各服务回滚步骤

### 4.1 总则

1. **决策确认**：白帆下达回滚指令（或 R-6 人工裁决）。
2. **流量止血**：将入口流量切至维护页或 100% LKG 实例（视流量方案：反向代理权重 / K8s rollout undo / compose 切 tag）。
3. **回滚顺序**（与起栈相反，前端先断流）：

```
web → gateway → insight → core →（按需）DB
```

4. **依赖不回滚**：PostgreSQL / Redis / NATS / MinIO **保持运行**，仅切换服务镜像与配置。
5. **恢复流量**：验证 §5 通过后逐步恢复。

### 4.2 分服务操作

#### 4.2.1 web

| 步骤 | 操作 | 验证 |
|---|---|---|
| W-1 | 将 `smartdesk-web` 部署切回 LKG 镜像 tag | Pod/容器 Running |
| W-2 | 确认 `NEXT_PUBLIC_API_BASE` 仍指向 gateway `/api/v1` | 环境变量与 LKG 快照一致 |
| W-3 | 若有静态资源 CDN 缓存，刷新关键路径 | — |
| W-4 | `GET /healthz` → 200 | liveness 通过 |

#### 4.2.2 gateway

| 步骤 | 操作 | 验证 |
|---|---|---|
| G-1 | 切回 LKG 镜像；恢复 LKG 配置文件（Redis 地址、mTLS 证书、下游 core/insight 地址） | — |
| G-2 | 滚动重启；等待旧实例排空 | 无 in-flight 502 尖峰 |
| G-3 | `GET /healthz` → 200；`GET /readyz` → 200 | Redis 连通 + 下游可达 |
| G-4 | 抽检 `POST /api/v1/auth/login`（测试账号） | 200 + JWT |

> gateway 为**唯一对外入口**；回滚后须确认限流规则与 LKG 一致，避免误封。

#### 4.2.3 insight

| 步骤 | 操作 | 验证 |
|---|---|---|
| I-1 | 切回 LKG 镜像；恢复 LKG 配置（DB DSN、NATS、SMTP 等） | — |
| I-2 | 滚动重启 worker + api 进程 | — |
| I-3 | `GET /healthz` → 200；`GET /readyz` → 200 | DB + NATS 连通（insight 探测；core 不强制） |
| I-4 | 观测 consumer lag 下降 | `/metrics` lag < 基线 |

> insight 故障**不阻塞建单**（降级红线）；但若新版本导致 consumer 崩溃，回滚可消除事件堆积。

#### 4.2.4 core

| 步骤 | 操作 | 验证 |
|---|---|---|
| C-1 | 切回 LKG 镜像；恢复 LKG 配置（DB DSN、NATS、OSS） | — |
| C-2 | 滚动重启；outbox relay 自动恢复 | — |
| C-3 | `GET /healthz` → 200；`GET /readyz` → 200 | 进程存活 + 服务可接收流量；DB/NATS 由独立基础设施探针覆盖 |
| C-4 | 检查 outbox 堆积 | lag 指标趋于平稳 |

> core 为**权威状态源**；回滚窗口内避免执行写操作直至 C-3 通过。

### 4.3 数据库迁移逆向策略

core 使用 `golang-migrate`，迁移文件 `migrations/NNNN_<name>.{up,down}.sql`（详见 core 子系统详设 §3）。

| 场景 | 策略 | 执行人 | 风险 |
|---|---|---|---|
| **S1：仅 bugfix，无 schema 变更** | 不回滚 DB | — | 低 |
| **S2：有 schema 变更，且 down 脚本已评审** | 按依赖逆序执行 `migrate down` 至 LKG 版本 | 石磊（core）+ 白帆批准 | 中：可能丢失新列数据 |
| **S3：有 schema 变更，down 未验证或不可逆** | **仅回滚镜像，保留 schema**；若 LKG 代码不兼容，升级为 S2 或热修 | 石磊 + 梁栋裁决 | 高：需评估兼容性 |

**默认策略：S1 或 S3**（不执行 down），优先保障 15 min RTO。

insight 使用 Alembic，原则相同：默认不回滚 schema，除非白帆 + 苏睿确认 down 安全。

### 4.4 一键回滚命令模板（docker-compose 环境）

> 生产环境由白帆维护的 CI/CD 流水线封装等价操作；以下为集成/预发环境参考。

```bash
# 0. 变量（由发布记录注入）
export LKG_TAG="v0.2.0-m2"

# 1. 切镜像 tag 并滚动重启（顺序：web → gateway → insight → core）
docker compose stop web
docker compose up -d --no-deps web    # compose 文件已指向 LKG_TAG

docker compose up -d --no-deps gateway
docker compose up -d --no-deps insight
docker compose up -d --no-deps core

# 2. （可选，仅 S2 场景）DB 回滚 — 需石磊现场确认
# docker compose exec core migrate -path /migrations -database "$DSN" down 1

# 3. 健康检查
for svc in core insight gateway web; do
  curl -sf "http://localhost:${PORT[$svc]}/readyz" || echo "FAIL: $svc"
done
```

## 5. 回滚后验证清单

### 5.1 P0 冒烟（必过，≤ 5 min）

| # | 验证项 | 方法 | 通过标准 |
|---|---|---|---|
| V-01 | 四服务就绪 | `GET /healthz` + `/readyz` × 4 | 全部 200 |
| V-02 | 登录 | `POST /api/v1/auth/login` | 200 + 有效 JWT |
| V-03 | 建单（核心路径） | `POST /api/v1/tickets` | **201**，status=`new`，不等待 AI |
| V-04 | 工单查询 | `GET /api/v1/tickets/{id}` | 200，字段完整 |
| V-05 | 状态流转 | `POST …/transitions`（accept→in_progress） | 200；非法跃迁仍 409 |
| V-06 | 越权红线 | requester 访问他人工单 | **403** |
| V-07 | 指标恢复 | Prometheus：5xx 率、P95 | 回到 §2.1 基线以内 |
| V-08 | 事件 lag | consumer lag | < 500 且不再攀升 |

### 5.2 P1 增强（≤ 15 min 内完成，可并行）

| # | 验证项 | 对应集成场景 |
|---|---|---|
| V-09 | AI 降级仍建单成功 | S-02：停 insight 后建单仍 201 |
| V-10 | 附件上传/下载 | S-08 |
| V-11 | SLA 计时可见 | S-09 |
| V-12 | 通知中心可读 | S-14（站内） |
| V-13 | 全链路 trace | `trace_id` 贯穿 gateway→core |

### 5.3 不通过处理

- **P0 任一失败**：维持流量隔离，升级石磊/关山/苏睿/江颜联合排查；白帆决定是否二次回滚（更早 LKG）或热修。
- **P0 全过、P1 部分失败**：可恢复流量，失败项建 issue 跟踪（不阻塞回滚闭环）。

## 6. 决策与通报链路

### 6.1 角色与职责

| 角色 | 负责人 | 职责 |
|---|---|---|
| **回滚决策** | 白帆（发布经理） | Go/No-Go 裁决；升级 CTO / 人类 |
| **回滚执行** | 万全 | 按本 Runbook 执行、采集观测数据 |
| **DB 迁移** | 石磊（core）/ 苏睿（insight） | S2/S3 场景技术裁决与操作 |
| **安全确认** | 武安 | R-5 安全事件判定 |
| **架构支持** | 梁栋 | 契约/跨服务兼容性裁决 |
| **人类通知** | CTO → `multica-test@hotmail.com` | 重大回滚、数据丢失风险 |

### 6.2 通报流程

```
观测异常 → 万全采集快照 → 通报白帆（≤3min）
    ↓
白帆裁决：暂停 / 回滚 / 继续观测
    ↓（回滚）
万全执行 §4 → 执行 §5 P0 验证
    ↓
白帆确认恢复 → 通报：
  - 发布运维群（白帆、万全）
  - 模块 Leader（石磊、关山、苏睿、江颜）
  - 测试（严谨）— 安排回归
  - 重大事故 → CTO → 人类邮件
```

### 6.3 通报模板

```text
【SmartDesk 回滚通报】
时间：<UTC>

自版本：<bad_tag> → LKG：<lkg_tag>
执行人：万全
状态：<执行中 / 已恢复 / 未恢复>
P0 验证：<通过项列表 / 失败项>
影响：<用户影响评估>
```
