## 1. 产品概述与目标

### 1.1 一句话定义
SmartDesk 是面向**企业内部**的、**AI 辅助**的智能工单 / 服务台系统：员工一键提单，系统自动分类定级、推荐相似工单，坐席协同处理，管理者实时掌握 SLA 与质量数据。

### 1.2 要解决的问题
| 痛点 | 现状 | SmartDesk 目标 |
|---|---|---|
| 提单门槛高、描述不规范 | 邮件 / IM 散乱，信息缺失 | 结构化一键提单 + 智能补全分类 |
| 分类、定级靠人工、易错漏 | 坐席手工打标、定优先级 | AI 自动分类 / 定级（人可纠偏） |
| 重复问题反复处理 | 同类问题重复排查 | 相似工单 / 历史方案推荐 |
| 处理过程不透明、SLA 失控 | 缺乏统一状态与时限管理 | 工单全生命周期 + SLA 计时与预警 |
| 管理者看不到全局质量 | 无统一看板 | 实时 SLA / 积压 / 满意度看板 |

### 1.3 成功度量（产品级，待人类确认目标值，见 §10 OQ-1）
- 提单到首次响应（FRT）中位数下降；SLA 达成率 ≥ 目标值。
- AI 自动分类**采纳率**与**准确率**达到约定阈值。
- 相似工单推荐对解决时长（MTTR）的缩短幅度。
- 坐席单位工时处理工单数提升；用户满意度（CSAT，若纳入）。

## 2. 用户角色与权限（RBAC）

> 权限由网关（`smartdesk-gateway`）统一鉴权，按角色（Role）授予能力，最小权限原则。一个账号可兼多角色。

| 角色 | 代号 | 关键权限 | 说明 |
|---|---|---|---|
| 报单人 / 员工 | `requester` | 提单、查看与追加自己的工单、评论、确认解决、重开、评价 | 全员默认角色 |
| 坐席 / 客服 | `agent` | 受理、处理、内部备注、转派、关联 / 合并、上传附件、流转状态 | 一线处理 |
| 坐席组长 / 分派人 | `lead` | 分派 / 改派、跨人查看本组工单、升级、调整优先级、管理 SLA 策略（本组） | 管理一个坐席组 |
| 管理者 | `manager` | 全局只读看板与报表、导出、查看质量指标 | 不直接处理工单 |
| 系统管理员 | `admin` | 用户 / 角色 / 权限、分类体系、SLA 策略、通知策略、系统配置 | 平台运维配置 |

> RBAC 资源 / 动作矩阵的完整定义在用户故事文档与架构设计阶段细化；本表为能力边界基线。

## 3. 范围边界：In / Out

### 3.1 第一阶段 In Scope（MVP + 必备）
- 账号认证（登录 / 登出 / 会话）、JWT、RBAC、网关统一入口与限流。
- 工单全生命周期：创建、分类、定级、分派、状态流转、评论、内部备注、附件、关联 / 合并、SLA 计时与预警、重开 / 关闭。
- 智能分析：自动分类、优先级建议、相似工单推荐、基础统计聚合。
- 通知与事件：领域事件总线 + 站内通知 + 邮件通知 + SLA 即将超时 / 已超时提醒。
- 协作：分派 / 改派、转派、升级、@提及内部协作。
- 看板与报表：SLA 达成、积压、坐席工作量、（可选）满意度等关键视图。
- 三端 Web：报单人门户、坐席工作台、管理后台。

### 3.2 Out of Scope（本阶段不做 / 待定，见 §10）
- 移动端原生 App（仅响应式 Web）。
- 邮件 / IM（企业微信 / 钉钉 / Slack）**入站**自动建单（OQ-3）。
- 知识库 / 自助 FAQ 门户（OQ-6）。
- 多租户 / 多组织隔离（OQ-7）。
- 计费、合同、资产管理等非服务台业务。
- 复杂 BI / 自助报表设计器（仅提供预置看板）。

## 4. 功能范围与能力清单

> 每个能力对应一组用户故事（见配套文档的 US 编号）。

### F1 认证与权限（gateway）
- F1.1 登录 / 登出、会话与令牌刷新（JWT）。
- F1.2 RBAC：角色与权限校验，未授权 403。
- F1.3 网关统一路由聚合、限流、审计日志埋点。
- F1.4 身份来源对接（SSO / 目录，见 OQ-2）。

### F2 工单生命周期（core）
- F2.1 结构化提单（标题、描述、分类、附件、关联资产 / 服务项）。
- F2.2 状态机：`新建 → 受理 → 处理中 → 待用户 → 已解决 → 已关闭`，支持 `重开`、`挂起`、`取消`（最终状态机以 §10 OQ-8 确认为准）。
- F2.3 分派 / 改派 / 自动分派（按分类 / 负载 / 技能组）。
- F2.4 评论（对外可见）与内部备注（仅坐席可见）。
- F2.5 附件上传 / 下载（大小 / 类型限制见 OQ-9）。
- F2.6 工单关联与合并（重复单合并）。
- F2.7 SLA 计时：按优先级设响应 / 解决时限，暂停（待用户）与恢复，超时标记。
- F2.8 工单时间线 / 操作审计（who-when-what）。

### F3 智能分析（insight）
- F3.1 自动分类：根据标题 / 描述预测分类，置信度 ≥ 阈值时自动填充，否则给建议（OQ-4）。
- F3.2 优先级 / 定级建议。
- F3.3 相似工单 / 历史解决方案推荐（OQ-5：检索方案选型）。
- F3.4 统计聚合：按时间 / 分类 / 坐席 / 优先级出量、SLA、时长等指标供看板消费。

### F4 通知与事件（insight / event 能力）
- F4.1 领域事件：建单、分派、状态变更、评论、SLA 预警 / 超时、解决、关闭等发布到事件总线。
- F4.2 站内通知中心（未读 / 已读）。
- F4.3 邮件通知（关键事件，模板化）。
- F4.4 通知策略：按角色 / 事件类型可配置订阅（admin）。

### F5 协作（core + web）
- F5.1 转派、升级（上报组长 / 管理者）。
- F5.2 @提及触发通知。
- F5.3 工单跟随 / 关注。

### F6 看板与报表（web + insight）
- F6.1 坐席个人工作台：我的待办、即将超时、按状态分组。
- F6.2 组长 / 管理者看板：SLA 达成率、积压趋势、坐席工作量、分类分布。
- F6.3 质量视图：重开率、超时单、（可选）CSAT。
- F6.4 数据导出（CSV）。

### F7 三端前端（web）
- F7.1 报单人门户（提单 + 我的工单）。
- F7.2 坐席工作台（队列 + 处理）。
- F7.3 管理后台（分类体系、SLA 策略、用户 / 角色、通知策略配置）。

## 5. 核心业务流程

### 5.1 提单到关闭主流程
```
报单人提单
  → 网关鉴权(F1) → core 建单(F2.1)
  → insight 自动分类/定级/相似推荐(F3.1~F3.3) → 写回建议
  → core 自动/手工分派(F2.3) → SLA 计时启动(F2.7)
  → 坐席受理→处理(F2.4/F2.5)，必要时转派/升级(F5)
  → [待用户] 时 SLA 暂停；用户回复后恢复
  → 坐席置「已解决」→ 通知报单人(F4)
  → 报单人确认→「已关闭」(可评价)；或在重开窗口内重开(F2.2)
全程领域事件落总线(F4.1)，看板实时消费(F6)
```

### 5.2 SLA 超时预警流程
```
SLA 计时器(F2.7) → 临近阈值 → 发布"SLA 即将超时"事件(F4.1)
  → 通知坐席+组长(F4.2/F4.3) → 超时则标记+升级建议(F5)
```

## 6. 模块 / 能力划分初稿（供架构团队输入）

> 这是**产品视角**的能力到代码仓的初步映射，作为架构团队（梁栋 / 秦诺）做总体架构与契约设计的输入。**最终服务边界、数据归属与契约以架构团队设计为准。**

| 代码仓 | 服务 | 语言 | 承载能力（PRD 编号） | 关键对外契约（待架构细化） |
|---|---|---|---|---|
| `smartdesk-web` | 前端 Web | TS + React/Next.js | F7（门户 / 工作台 / 后台）、F6 看板 UI、F2/F5 交互 | 调用 gateway 聚合 API |
| `smartdesk-gateway` | 网关 / 认证 BFF | TS + Node/NestJS | F1 全部、路由聚合、限流、审计 | 对外统一 REST/BFF；向后端服务转发 |
| `smartdesk-core` | 工单核心 | Go | F2 全部、F5 协作 | 工单 CRUD/状态机 API、领域事件发布、附件 |
| `smartdesk-insight` | 智能分析 | Python/FastAPI | F3 全部、F4 通知与事件消费 / 分发、F6 指标聚合 | 分类 / 相似 / 统计 API；事件订阅；通知发送 |

**跨切关注点（需架构统一定义）**：
- 领域事件模型与事件总线选型（同步调用 vs 消息队列）。
- 统一鉴权令牌在服务间的传递与服务间信任。
- 分类体系（taxonomy）与 SLA 策略的**配置数据归属**（建议归 core 或独立配置，待定）。
- 数据存储划分（工单 OLTP、分析 / 检索存储、附件对象存储）。
- 相似检索的技术路线（关键词 / 向量；模型与向量库选型，OQ-5）。
- 可观测性：日志 / 指标 / 链路追踪基线。

**首批待定义契约（PRD 建议清单，供架构 OpenAPI 设计）**：建单、查询 / 列表（过滤 / 分页）、状态流转、分派、评论 / 备注、附件、SLA 查询、分类预测、相似推荐、统计聚合、事件 schema、通知发送。

## 7. 非功能需求（NFR）

| 类别 | 要求（基线，目标值待 §10 确认） |
|---|---|
| 性能 | 列表 / 详情 P95 响应 < 500ms（常规负载）；AI 分类 / 相似建议 P95 < 2s，且**不阻塞**建单主流程（异步写回） |
| 可用性 | 目标可用性 ≥ 99.5%（阶段目标，OQ-1）；核心建单链路具备降级（AI 不可用时建单仍成功） |
| 可扩展 | 微服务可独立部署 / 扩缩容；事件驱动解耦 |
| 安全 | 遵循 OWASP ASVS / Top10；强制鉴权与 RBAC；输入校验；越权防护；敏感数据加密传输（TLS）；审计日志 |
| 合规 / 隐私 | PII 最小化与留存策略（OQ-10）；附件访问鉴权 |
| 可靠性 | 工单状态变更幂等；关键操作可审计、可追溯；通知至少一次送达 |
| 可观测 | 统一日志、关键指标、健康检查 |
| 国际化 | 第一阶段 UI 语言范围见 OQ-11（默认中文） |
| 可维护 | 契约先行（OpenAPI）；关键代码集体检视门禁（见组织说明书 §11） |
| 兼容 | 主流现代浏览器（Chrome/Edge/Firefox/Safari 最新两个大版本） |

## 8. 验收标准基线

PRD 级验收遵循三层：

1. **功能验收**：每条用户故事的 Given/When/Then 验收标准全部通过（见配套文档），由测试团队（严谨 / 林桥）验证。
2. **NFR 验收**：性能、可靠性、可用性按 §7 目标值（人类确认后）验证；安全测试（武安）对照 OWASP ASVS/Top10、组件安全基线，网关认证 / RBAC 必过越权与鉴权用例。
3. **业界规范基线**：对照《组织说明书》§10.3 规范基线清单（安全 / 可靠性 / 性能 / 代码质量），由质量管理团队维护清单、测试团队据此验证。

**PRD 自身的"完成"判据**：功能范围、角色权限、用户故事 + 验收标准、模块划分初稿、NFR、开放问题清单齐备，经温知把关、CTO 评审，**人类逐条裁决开放问题并确认冻结**。

## 11. 术语表

| 术语 | 含义 |
|---|---|
| 工单 / Ticket | 一次服务请求的记录载体 |
| SLA | 服务级别协议：响应 / 解决时限 |
| FRT | First Response Time，首次响应时间 |
| MTTR | 平均解决时长 |
| 坐席 / Agent | 处理工单的客服人员 |
| RBAC | 基于角色的访问控制 |
