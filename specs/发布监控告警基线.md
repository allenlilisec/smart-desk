## 1. 目标与范围

为 SmartDesk **四服务**（gateway / core / insight / web）在 M4 灰度上线前建立统一监控告警基线，支撑：

1. **金丝雀放量决策**：每批 5% / 25% / 100% 的 Go/No-Go 观测窗口；
2. **回滚触发**：P0 告警联动一键回滚 Runbook（目标 RTO ≤ 15 min）；
3. **日常运维**：staging / production 服务健康与关键业务 SLI 持续可见。

**不在本稿范围**：具体 Prometheus/Grafana/Alertmanager 部署清单（由白帆 CI/CD 流水线 issue 落地）；告警规则 YAML 实现（本稿为**阈值与语义基线**，实现随 staging 环境就绪后提交）。

## 2. 服务清单与健康检查

| 服务 | 语言/框架 | Liveness | Readiness 探测项 | 指标端点 | 备注 |
|---|---|---|---|---|---|
| **gateway** | TS/NestJS | `GET /healthz` | Redis 连通 + core/insight 下游可达 | `GET /metrics` | 对外入口；鉴权/限流在此收口 |
| **core** | Go | `GET /healthz` | 服务本身可接收流量（不强制 DB/NATS） | `GET /metrics` | 内网；建单/状态机权威；依赖健康由独立基础设施探针覆盖 |
| **insight** | Py/FastAPI | `GET /healthz` | PostgreSQL 读模型 + NATS | `GET /metrics` | 内网；AI/通知/统计 |
| **web** | TS/Next.js | `GET /healthz` | （静态路由，无外部依赖） | — | 前端；通过 gateway BFF 访问 API |

### 2.1 健康检查约定（对齐 `service-scaffold`）

| 端点 | 语义 | 成功 | 失败 | 采集频率 |
|---|---|---|---|---|
| `/healthz` | 进程存活 | HTTP 200 | 非 200 或超时 | K8s liveness：10s interval，3 次失败重启 |
| `/readyz` | 可接流量 | HTTP 200 | HTTP 503 | K8s readiness：5s interval，连续 2 次失败摘流 |

**web** 仅暴露 `/healthz`（详设约定）；就绪性通过 gateway `/readyz` 及端到端冒烟间接验证。

### 2.2 基础设施探针（并行采集，不替代服务 `/readyz`）

| 组件 | 探针 | 失败语义 |
|---|---|---|
| PostgreSQL | `pg_isready` / 连接池活跃连接 | insight readiness 503；core readiness 不因此失败 |
| Redis | `PING` | gateway readiness 503 |
| NATS JetStream | 连接 + stream 存在性 | core、insight readiness 503 |
| MinIO/OSS | bucket HEAD | core 附件链路降级（不阻塞建单） |

## 3. 各服务必采指标

### 3.1 通用 HTTP 指标（gateway / core / insight）

所有 HTTP 服务通过 middleware 暴露 Prometheus 指标，标签至少含：`service`、`method`、`route`（路由模板，非原始 path）、`status_code`。

| 指标名（建议） | 类型 | 说明 |
|---|---|---|
| `http_requests_total` | Counter | 请求总量 |
| `http_request_duration_seconds` | Histogram | 延迟分布；桶：0.005, 0.01, 0.025, 0.05, 0.1, 0.25, 0.5, 1, 2.5, 5, 10 |
| `http_requests_in_flight` | Gauge | 当前处理中请求数 |

**派生 SLI**（所有服务）：

| SLI | PromQL 示意 | 说明 |
|---|---|---|
| 请求量 QPS | `sum(rate(http_requests_total{service="X"}[5m]))` | 按服务/路由聚合 |
| 错误率 | `sum(rate(http_requests_total{service="X",status_code=~"5.."}[5m])) / sum(rate(http_requests_total{service="X"}[5m]))` | 5xx 占比 |
| P50 延迟 | `histogram_quantile(0.50, sum(rate(http_request_duration_seconds_bucket{service="X"}[5m])) by (le))` | |
| P95 延迟 | `histogram_quantile(0.95, ...)` | gateway 列表/详情 NFR < 500ms |
| P99 延迟 | `histogram_quantile(0.99, ...)` | |

### 3.2 gateway 补充指标

| 指标名（建议） | 类型 | 说明 |
|---|---|---|
| `gateway_auth_attempts_total` | Counter | 标签：`result`（success/failure/locked） |
| `gateway_auth_failures_total` | Counter | 标签：`reason`（invalid_credentials/expired/revoked） |
| `gateway_rbac_denied_total` | Counter | 403 越权拒绝 |
| `gateway_rate_limit_exceeded_total` | Counter | 429 限流 |
| `gateway_upstream_errors_total` | Counter | 标签：`upstream`（core/insight）、`status_code` |
| `gateway_session_active` | Gauge | Redis 活跃会话数 |

### 3.3 core 补充指标

| 指标名（建议） | 类型 | 说明 |
|---|---|---|
| `core_tickets_created_total` | Counter | 标签：`result`（success/failure）、`priority` |
| `core_ticket_transitions_total` | Counter | 标签：`from_status`、`to_status`、`result` |
| `core_sla_timer_accuracy_seconds` | Histogram | SLA 计时偏差（系统详设 §10） |
| `core_outbox_lag_seconds` | Gauge | 事件 outbox 投递滞后 |
| `core_event_consumer_lag_seconds` | Gauge | 标签：`event_type`；消费 `insight.classification_suggested` 等 |
| `core_idempotency_conflicts_total` | Counter | Idempotency-Key 冲突 409 |

### 3.4 insight 补充指标

| 指标名（建议） | 类型 | 说明 |
|---|---|---|
| `insight_classification_requests_total` | Counter | 标签：`result`（success/degraded/503） |
| `insight_classification_duration_seconds` | Histogram | 分类/定级耗时；NFR P95 < 2s |
| `insight_event_consumer_lag_seconds` | Gauge | 标签：`stream`、`consumer` |
| `insight_events_processed_total` | Counter | 标签：`event_type`、`result` |
| `insight_classification_adoption_rate` | Gauge | 分类建议采纳率（系统详设 §10） |
| `insight_notification_dead_letter_total` | Counter | 通知死信（M4 加固项） |

### 3.5 web 补充指标

| 指标名（建议） | 类型 | 说明 |
|---|---|---|
| `web_page_load_duration_seconds` | Histogram | 关键页面（报单/工作台） |
| `web_api_client_errors_total` | Counter | 前端观测到的 gateway 4xx/5xx |
| `web_js_errors_total` | Counter | 客户端未捕获异常 |

> web 指标通过 RUM/前端上报或合成探针采集；`/metrics` 端点非必须，但 `/healthz` 为硬性要求。

## 4. 关键业务 SLI

以下 SLI 为灰度 Go/No-Go 与告警的**业务视角**核心指标，优先级高于单服务通用 HTTP 指标。

| SLI ID | 名称 | 定义 | 数据源 | NFR/基线参考 |
|---|---|---|---|---|
| **SLI-01** | 建单成功率 | `成功建单数 / 建单请求数`（排除 4xx 客户端错误） | gateway `POST /tickets` 或 core `core_tickets_created_total{result="success"}` | ≥ 99.9%（灰度窗口）；核心链路可降级：insight 不可用仍建单成功 |
| **SLI-02** | 鉴权失败率 | `鉴权失败数 / 鉴权尝试数` | `gateway_auth_failures_total / gateway_auth_attempts_total` | 突增 > 基线 3σ 或绝对值 > 5%（5 min）→ 调查；> 20% → P0 |
| **SLI-03** | 事件消费滞后 | `now() - last_successful_consume_timestamp` 或 `*_consumer_lag_seconds` | core `core_event_consumer_lag_seconds`、insight `insight_event_consumer_lag_seconds` | P95 < 30s（常规）；灰度窗口 < 60s |
| **SLI-04** | 入口可用性 | gateway `/readyz` 成功率 | 黑盒探针 | ≥ 99.5%（PRD §7） |
| **SLI-05** | 读路径延迟 | gateway 列表/详情 P95 | `http_request_duration_seconds{route=~"/tickets.*"}` | P95 < 500ms |
| **SLI-06** | AI 建议延迟 | 分类写回端到端 P95 | insight 消费 lag + 写回可见性 | P95 < 2s，且不阻塞建单（红线） |
| **SLI-07** | 越权拒绝率 | RBAC 403 占比 | `gateway_rbac_denied_total` | 异常突增需安全协同（非单纯容量问题） |

### 4.1 建单成功率（SLI-01）细化

```
建单成功率 = 
  sum(rate(core_tickets_created_total{result="success"}[5m]))
  /
  sum(rate(http_requests_total{service="gateway",route="/tickets",method="POST"}[5m]))
```

- **分母排除**：400（校验）、401（未登录）、409（幂等冲突）、413（附件过大）—— 计入客户端错误，不降低成功率判定。
- **灰度对比**：新版本实例与稳定版本分别打 `version` 标签，按流量比例加权对比。

### 4.2 鉴权失败率（SLI-02）细化

- 正常背景：登录失败（错误密码）会产生失败计数，需按 `reason` 分桶。
- **告警关注**：`invalid_credentials` 突增（撞库）或 `expired` 批量出现（时钟/密钥问题）。
- 与 US-1.3 限流 429 区分：429 不计入鉴权失败。

### 4.3 事件消费滞后（SLI-03）细化

| 消费方 | 关键 stream / event | 滞后定义 |
|---|---|---|
| insight | `ticket.created` 等 core 事件 | 最新消息消费时间落后发布时间 |
| core | `insight.classification_suggested` | 写回建议可见性；M3+ 启用 |
| core outbox relay | outbox → NATS | `core_outbox_lag_seconds` |

**分区语义**（D3）：单 `ticket_id` 内有序；跨工单不保证全局序。告警按 consumer group 聚合 lag，不按单分区抖动。

## 5. 告警分级与响应动作

| 级别 | 名称 | 响应时限 | 自动/人工动作 | 典型触发条件（v0.1 草案阈值，**待白帆确认**） |
|---|---|---|---|---|
| **P0** | 立即回滚 | 5 min 内决策，15 min 内完成回滚 | 万全执行回滚 Runbook；通知白帆 + CTO（人类） | 见 §5.1 |
| **P1** | 暂停放量 | 10 min 内 | 停止金丝雀流量推进；白帆组织研判 | 见 §5.2 |
| **P2** | 观察 | 30 min 内 | 记录、趋势跟踪；下一观测窗口复核 | 见 §5.3 |

### 5.1 P0 — 立即回滚

满足**任一**条件即触发（与灰度策略联动，见 §7）：

| 规则 ID | 条件 | 持续 |
|---|---|---|
| P0-01 | 任意服务 `/readyz` 成功率 < 95% | 2 min |
| P0-02 | gateway 5xx 错误率 > 5% | 3 min |
| P0-03 | 建单成功率（SLI-01）< 95% | 3 min |
| P0-04 | 鉴权失败率（SLI-02）> 20%（排除计划内密钥轮换窗口） | 2 min |
| P0-05 | 事件消费滞后（SLI-03）> 300s 且持续增长 | 5 min |
| P0-06 | gateway 读路径 P95 > 2s | 5 min |
| P0-07 | 核心链路降级失败：insight/NATS 故障时建单仍失败 | 即时 |

### 5.2 P1 — 暂停放量

满足**任一**条件即触发；**不自动回滚**，等待白帆 + 万全研判：

| 规则 ID | 条件 | 持续 |
|---|---|---|
| P1-01 | gateway 5xx 错误率 1%–5% | 5 min |
| P1-02 | 建单成功率 95%–99.9% | 5 min |
| P1-03 | 鉴权失败率 5%–20% | 5 min |
| P1-04 | 事件消费滞后 60s–300s | 5 min |
| P1-05 | gateway 读路径 P95 500ms–2s | 10 min |
| P1-06 | insight 分类 P95 2s–5s（建单不受影响） | 10 min |
| P1-07 | 金丝雀批次内新版本错误率比稳定版高 > 2x | 5 min |

### 5.3 P2 — 观察

| 规则 ID | 条件 | 说明 |
|---|---|---|
| P2-01 | 错误率 / 延迟较基线偏离 20%–50% | 未达 P1 门槛 |
| P2-02 | 事件消费滞后 30s–60s | 常规波动上限 |
| P2-03 | 分类采纳率较 7 日基线下降 > 10% | M3+ 智能增强阶段 |
| P2-04 | outbox lag > 10s | 关注 relay 健康 |

## 7. 与灰度策略观测窗口的对齐

### 7.1 金丝雀分批策略（`deploy-canary`）

| 批次 | 流量比例 | 最短观测窗口 | 进入下一批条件 |
|---|---|---|---|
| C1 | 5% | **15 min** | P0=0、P1=0；SLI-01 ≥ 99.9%；SLI-03 < 60s |
| C2 | 25% | **30 min** | 同上 + SLI-05 P95 < 500ms |
| C3 | 100% | **60 min** | 同上；**CTO 人类确认**（与 [`灰度发布策略.md`](灰度发布策略.md) §7 G5 同句同角色） |

**异常即暂停**：任一 P0 → 立即回滚至上一稳定版本；任一 P1 → 冻结当前批次，不推进流量。

### 7.2 观测窗口内必查仪表盘

每批放量后，万全在观测窗口内每 **5 min** 记录以下快照：

| 面板 | 指标 |
|---|---|
| 服务健康 | 四服务 `/healthz`、`/readyz` 成功率 |
| 黄金信号 | QPS、错误率、P50/P95/P99 延迟（gateway + core） |
| 业务 SLI | SLI-01 建单成功率、SLI-02 鉴权失败率、SLI-03 消费滞后 |
| 金丝雀对比 | 新版本 vs 稳定版错误率/延迟（按 `version` 标签） |
| 依赖 | PG/Redis/NATS 探针状态 |

### 7.3 灰度 vs 告警阈值双向对齐

| 灰度门禁 | 对应告警规则 | 对齐说明 |
|---|---|---|
| C1/C2/C3 要求 SLI-01 ≥ 99.9% | P1-02（< 99.9%）、P0-03（< 95%） | 灰度门禁严于 P1，P0 为灾难性下跌 |
| C1/C2 要求 SLI-03 < 60s | P1-04（60–300s）、P0-05（> 300s） | 灰度窗口用 60s，告警分级递增 |
| C2+ 要求 P95 < 500ms | P1-05、P0-06 | 与 PRD NFR 一致 |
| 异常即暂停 | P1 触发即冻结批次 | 技能语义与 P1 动作一致 |
| 人类确认 100% | 非告警；流程门禁 | 告警清零 ≠ 自动放量 |

### 7.4 回滚后观测

回滚完成后 **15 min** 内持续观测：确认 SLI 恢复至回滚前稳定基线 ±10%；若未恢复，升级 P0 为基础设施/数据层故障，通知石磊 + 白帆。

## 附录 A：指标命名与 `service-scaffold` 对齐

各服务脚手架（`service-scaffold` 技能）须默认暴露：

- `GET /healthz`、`GET /readyz`（web 仅 `/healthz`）
- `GET /metrics`（Prometheus 文本格式；web 可选）
- HTTP middleware 自动打点 `http_requests_total`、`http_request_duration_seconds`
- 日志字段 `trace_id` 与 OTel trace 贯通

本基线指标名与标签为**推荐约定**；实现时可沿用脚手架默认名，但须在 staging 仪表盘映射到本文 SLI 定义。
