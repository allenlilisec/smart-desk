## 1. 目标与范围

为 SmartDesk **四服务**（gateway / core / insight / web）在 M4 上线前建立统一金丝雀灰度发布策略，支撑：

1. **分批放量**：C1 5% → C2 25% → C3 100%，每批最短观测窗口后人工确认推进；
2. **观测决策**：健康检查、错误率、P95 延迟、关键业务 SLI 达标方可进入下一批；
3. **暂停/回滚**：触发 H-* 暂停放量，触发 R-* 或 P0 执行回滚 Runbook（RTO ≤ 15 min）；
4. **人类确认**：G1–G5 五处门禁，**告警清零 ≠ 自动放量**，每批推进须 **CTO（Allen Li）** 人工确认。

**不在本稿范围**：具体 CI/CD job 实现（见 SUP-170）；Prometheus 规则 YAML（见监控基线 §8 staging 验证）。

## 2. 放量阶梯与观测窗口

| 批次 | 流量比例 | 最短观测窗口 | 进入下一批条件 |
|---|---|---|---|
| **C1** | 5% | **15 min** | P0=0、P1=0；SLI-01 ≥ 99.9%；SLI-03 < 60s；G2 CTO 确认 |
| **C2** | 25% | **30 min** | 同上 + SLI-05 P95 < 500ms；G3 CTO 确认 |
| **C3** | 100% | **60 min** | 同上；G4 CTO 确认后全量 |
| **收尾** | — | — | G5 晋升 LKG、关闭灰度窗口 |

**异常即暂停**：任一 P0 → 立即回滚；任一 P1 或 H-* → 冻结当前批次，不推进流量。

### 2.1 流量分割实现（OQ-CANARY-1 裁决）

| 环境 | 方案 | 说明 |
|---|---|---|
| **生产 K8s** | Ingress **权重路由** | 双 Deployment（stable/canary），Pod 标签 `version=stable\|canary` 供 SLI 对比 |
| **预发 compose** | **副本数比例**近似流量 | 如 1:19 ≈ 5%，供 D-01~D-06 dry-run；见 `deploy/docker-compose.canary.yml` |

## 3. 灰度策略选择

### 3.1 默认：四服务联动灰度

**默认采用四服务联动灰度**（gateway + core + insight + web 同步推进同一批次）。

**理由**：四服务通过契约、事件总线（NATS）、BFF 聚合强耦合；单服务金丝雀可能导致版本错配（如新 gateway 调旧 core 契约字段），观测结论不可信。

**执行**：同一 C1/C2/C3 批次内，四服务 canary 实例使用相同 `CANARY_TAG`，stable 实例使用相同 `STABLE_TAG`（LKG）。

### 3.2 例外：单服务热修复独立灰度

满足以下**全部**条件时，可走单服务独立灰度：

1. 白帆书面批准（issue 评论或发布记录）；
2. 契约 CI 通过（`api-contract-check` 无 breaking change）；
3. 变更范围不涉及跨服务事件 schema 或共享 DB migration；
4. 独立灰度仍须通过 G1 门禁与对应批次观测窗口。

## 4. 每批观测指标

每批放量后，万全在观测窗口内每 **5 min** 记录快照（对齐监控基线 §7.2）。

### 4.1 基础设施健康

| 指标 | 探测 | 通过标准 |
|---|---|---|
| Liveness | `GET /healthz` × 4 服务 | 全部 HTTP 200 |
| Readiness | `GET /readyz` × 3（web 仅 `/healthz`） | 全部 HTTP 200；连续 2 次失败即 H-3 |

### 4.2 黄金信号

| 指标 | 说明 | 金丝雀对比 |
|---|---|---|
| QPS | 各服务请求量 | 按 `version` 标签分桶 |
| 5xx 错误率 | HTTP 5xx 占比 | 新版本 vs 稳定版；P1-07：金丝雀错误率 > 稳定版 2x → 暂停 |
| P50/P95/P99 延迟 | gateway 聚合 + core 写路径 | 按 `version` 标签分桶 |

### 4.3 关键业务 SLI

| SLI ID | 名称 | 灰度门禁 | 数据源 |
|---|---|---|---|
| **SLI-01** | 建单成功率 | ≥ 99.9% | `core_tickets_created_total{result="success"}` / gateway `POST /tickets` |
| **SLI-02** | 鉴权失败率 | 无异常突增（见 §5 映射） | `gateway_auth_failures_total / gateway_auth_attempts_total` |
| **SLI-03** | 事件消费滞后 | < 60s（主指标：**秒级 lag**） | `*_consumer_lag_seconds` |
| **SLI-05** | 读路径 P95 | < 500ms（C2+） | gateway `http_request_duration_seconds{route=~"/tickets.*"}` |

## 5. 暂停/回滚阈值与告警映射

本策略 H/R 阈值与回滚预案 §2、监控基线 P0/P1 **双向对齐**。下表为**权威映射**（H/R ↔ P0/P1），执行时以先触发者为准。

### 5.1 暂停放量（Hold）— H-* ↔ P1

| H # | 策略阈值 | 对应 P1 | 映射说明 |
|---|---|---|---|
| **H-1** | 5xx 较基线上升 > 0.5% 或绝对值 > 1%（连续 2 min） | **P1-01**（1%–5%） | 灰度暂停阈值严于 P1 下沿；H-1 在 1% 即触发 |
| **H-2** | API P95 较基线恶化 > 30% 或 gateway 聚合 **> 800ms**（连续 3 min） | **P1-05**（500ms–2s） | **800ms 为灰度暂停早警**，与 P1-05 下沿（500ms）衔接 |
| **H-3** | 任一服务 `/readyz` 非 200（连续 2 次，间隔 15s） | **P0-01** 前兆 | readiness 失败即暂停；持续 2 min → P0-01 |
| **H-4** | NATS consumer lag **> 60s**（主指标）且持续上升（连续 5 min） | **P1-04**（60s–300s） | **主指标统一为秒级 lag**（`*_consumer_lag_seconds`，与 Prom 一致）；条数 lag > 5000 仅作辅助观测，不作为门禁主判据 |
| **H-5** | 金丝雀实例健康检查失败 | 即时 | 优先摘除金丝雀 Pod/容器 |

**动作**：停止下一批放量，万全采集快照，通报白帆研判。

### 5.2 强制回滚（Rollback）— R-* ↔ P0

| R # | 策略阈值 | 对应 P0 | 映射说明 |
|---|---|---|---|
| **R-1** | 5xx > 5% 或较基线上升 > 3% | **P0-02** | 一致 |
| **R-2** | 建单成功率 **< 99%** | **P0-03**（< 95%） | **R-2 先于 P0-03 触发**（安全侧更严）；R-2 触发即启动回滚，无需等待 P0-03 |
| **R-3** | 鉴权全面失效 | **P0-04**（> 20%） | **统一以 SLI-02 鉴权失败率为准**：`gateway_auth_failures_total / gateway_auth_attempts_total` > 20%（排除计划内密钥轮换窗口）；login 成功率 < 95% 为 SLI-02 的等价表述 |
| **R-4** | 数据一致性事故 | — | 需石磊确认 |
| **R-5** | 安全事件（越权可复现） | — | 武安确认后即时回滚 |
| **R-6** | 人工裁决（白帆 / CTO） | — | 优先级最高 |

> 灰度阶段若仅金丝雀实例异常且流量可隔离，优先**摘除金丝雀**；摘除非金丝雀后仍不达标，执行完整回滚（见回滚预案 §4）。

### 5.3 完整阈值对照速查

| 维度 | 灰度暂停（H/P1） | 强制回滚（R/P0） | 观测窗口 |
|---|---|---|---|
| 5xx 错误率 | > 1%（H-1）/ 1%–5%（P1-01） | > 5%（R-1/P0-02） | 2–5 min |
| 建单成功率 | 95%–99.9%（P1-02） | < 99%（R-2）→ < 95%（P0-03） | 3–5 min |
| 鉴权失败率（SLI-02） | 5%–20%（P1-03） | > 20%（R-3/P0-04） | 2–5 min |
| 消费滞后（秒） | > 60s（H-4/P1-04） | > 300s（P0-05） | 5 min |
| gateway P95 | > 800ms（H-2）/ 500ms–2s（P1-05） | > 2s（P0-06） | 3–10 min |

## 6. 人类确认点（G1–G5）

按 CTO 授权，灰度放量须 **CTO（Allen Li）** 人工确认；与监控基线 §7.1 C3「人类确认」**同句同角色**。

| 门禁 | 时机 | 确认人 | 前置条件 |
|---|---|---|---|
| **G1** | 灰度启动前 | CTO | LKG 清单冻结；四服务镜像构建通过；契约 CI 绿 |
| **G2** | C1（5%）→ C2（25%） | CTO | C1 观测窗口满 15 min；P0=0、P1=0；SLI 达标 |
| **G3** | C2（25%）→ C3（100%） | CTO | C2 观测窗口满 30 min；P0=0、P1=0；SLI-05 P95 < 500ms |
| **G4** | C3 全量确认 | CTO | C3 观测窗口满 60 min；全量 SLI 稳定 |
| **G5** | 晋升 LKG / 关闭灰度 | 白帆 + CTO | G4 通过；回滚窗口关闭；更新 LKG 清单 |

**告警清零 ≠ 自动放量**：P0/P1 清零后，万全整理观测报告，白帆提请 CTO 通过 `multica-test@hotmail.com` 或 Issue 评论确认下一批。

## 7. 执行流程

```
发布前：白帆冻结 LKG 清单 + 万全确认监控基线就绪
    ↓
G1：CTO 确认启动 → 部署 canary 实例（C1 5%）
    ↓
观测 15 min（§4 指标 + §5 阈值）→ P0/P1 清零？
    ├─ 否 → 暂停/回滚（§5）
    └─ 是 → G2 CTO 确认 → C2 25%
              ↓
         观测 30 min → G3 CTO 确认 → C3 100%
              ↓
         观测 60 min → G4 CTO 确认 → G5 晋升 LKG
```
