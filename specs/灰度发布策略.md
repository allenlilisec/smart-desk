## 1. 目标与范围

为 SmartDesk **四服务**（gateway / core / insight / web）在 M4 灰度上线阶段提供可执行的**金丝雀放量策略**，支撑：

1. **分批放量**：5% → 25% → 100% 阶梯推进，每批最短观测窗口内采集健康/错误率/延迟/业务 SLI；
2. **暂停/回滚门禁**：与 [`回滚预案.md`](回滚预案.md) §2 H/R 阈值及 [`发布监控告警基线.md`](发布监控告警基线.md) P0/P1 分级双向对齐；
3. **人类确认点**：按 CTO 授权，每批推进须 CTO 确认；**告警清零 ≠ 自动放量**（组织说明书 §8.19）。

**不在本稿范围**：CI/CD 流水线实现（白帆）；Prometheus 告警规则 YAML（staging 就绪后落地）。

---

## 2. 分批放量阶梯

| 批次 | 流量比例 | 最短观测窗口 | 进入下一批条件 |
|---|---|---|---|
| **C1** | 5% | **15 min** | P0=0、P1=0；SLI-01 ≥ 99.9%；SLI-03 < 60s |
| **C2** | 25% | **30 min** | 同上 + SLI-05 P95 < 500ms |
| **C3** | 100% | **60 min** | 同上；**CTO 人类确认**（见 §7 G5） |

**异常即暂停**：观测窗口内触发任一 H-* → 冻结当前批次、停止下一批放量；触发任一 R-* 或 P0 → 执行 [`回滚预案.md`](回滚预案.md) §4。

每批放量后，万全在观测窗口内每 **5 min** 记录仪表盘快照（详见 [`发布监控告警基线.md`](发布监控告警基线.md) §7.2）。

## 3. 四服务联动 vs 独立灰度

### 3.1 默认：四服务联动灰度

**默认策略为四服务联动灰度**（gateway / core / insight / web 同批同比例推进）。

| 理由 | 说明 |
|---|---|
| 契约耦合 | gateway BFF 与 core/insight OpenAPI 契约版本须一致 |
| 事件耦合 | core outbox → NATS → insight 消费链路跨服务 |
| 前端耦合 | web `NEXT_PUBLIC_API_BASE` 指向 gateway，API 行为变更需同步 |
| SLI 可比性 | 金丝雀对比（`version` 标签）要求上下游同版本 |

### 3.2 例外：单服务热修复独立灰度

满足**全部**条件时可走单服务独立灰度例外路径：

1. 白帆书面批准（issue 评论或发布记录）；
2. 契约 CI 通过（OpenAPI diff 无 breaking change）；
3. 变更范围限定为单服务 bugfix，不涉及跨服务事件 schema 变更；
4. 其余三服务保持 LKG 版本不动。

独立灰度仍须遵守本稿 §2 放量阶梯与 §5 阈值；观测范围扩展至该服务上下游依赖探针。

## 4. 每批观测指标

### 4.1 服务健康

| 指标 | 来源 | 通过标准 |
|---|---|---|
| Liveness | 四服务 `GET /healthz` | 100% 200 |
| Readiness | gateway/core/insight `GET /readyz` | 100% 200（web 经 gateway 间接验证） |
| 金丝雀探针 | 灰度实例健康检查 | 无失败（对应 H-5） |

### 4.2 黄金信号

| 指标 | 范围 | 说明 |
|---|---|---|
| 5xx 错误率 | gateway 聚合 | 与 LKG 基线对比 |
| P95 延迟 | gateway 聚合 + 读路径 | SLI-05 读路径 P95 |
| QPS | gateway | 流量比例校验（金丝雀 ≈ 目标比例） |

### 4.3 关键业务 SLI

| SLI ID | 名称 | 灰度门禁 | 数据源 |
|---|---|---|---|
| **SLI-01** | 建单成功率 | C1/C2/C3 均 ≥ 99.9% | `core_tickets_created_total{result="success"}` / gateway `POST /tickets` |
| **SLI-02** | 鉴权失败率 | 无突增（见 §6 映射） | `gateway_auth_failures_total / gateway_auth_attempts_total` |
| **SLI-03** | 事件消费滞后 | < 60s（**秒级**，见 §5 H-4） | `core_event_consumer_lag_seconds`、`insight_event_consumer_lag_seconds` |
| **SLI-05** | 读路径延迟 | C2+ P95 < 500ms | gateway `http_request_duration_seconds{route=~"/tickets.*"}` |

### 4.4 金丝雀对比

新版本与稳定版实例分别打 `version` 标签（canary / stable），按流量比例加权对比错误率与延迟。P1-07 规则：金丝雀批次内新版本错误率比稳定版高 > 2x → 暂停放量。

## 5. 暂停/回滚触发阈值

与 [`回滚预案.md`](回滚预案.md) §2 保持一致。满足**任一** H-* 即**暂停放量**；满足**任一** R-* 即**强制回滚**。

### 5.1 暂停放量（Hold）

| # | 指标 / 现象 | 阈值 | 观测窗口 | 备注 |
|---|---|---|---|---|
| H-1 | 5xx 错误率 | 较基线上升 **> 0.5%** 或绝对值 **> 1%** | 连续 2 min | 对齐 P1-01 下沿 |
| H-2 | API P95 延迟 | 较基线恶化 **> 30%** 或 gateway 聚合 **> 800ms** | 连续 3 min | **与 P1-05（500ms–2s）下沿衔接，800ms 为灰度暂停早警** |
| H-3 | `/readyz` 非 200 | 任一服务 readiness 失败 | 连续 2 次探测（间隔 15s） | 对齐 P0-01 前置 |
| H-4 | 事件消费 lag | **`*_consumer_lag_seconds` > 60s** 或持续上升 | 连续 5 min | **主指标为秒级 lag**（与 Prom 一致）；NATS consumer 条数 lag **> 5000** 仅作辅助观测，不作为暂停主判据 |
| H-5 | 金丝雀探针失败 | 灰度实例健康检查失败 | 即时 | — |

**动作**：停止下一批放量，万全采集快照（指标截图、日志 trace_id、镜像版本），通报白帆研判。

### 5.2 强制回滚（Rollback）

| # | 指标 / 现象 | 阈值 | 备注 |
|---|---|---|---|
| R-1 | 5xx 错误率 | **> 5%** 或较基线上升 **> 3%** | 对齐 P0-02 |
| R-2 | 核心建单失败 | `POST /api/v1/tickets` 成功率 **< 99%** | **先于 P0-03（< 95%）触发**；安全侧更严 |
| R-3 | 鉴权全面失效 | 见 §6.2 R-3 ↔ SLI-02 换算 | 对齐 P0-04 |
| R-4 | 数据一致性事故 | 状态机非法跃迁批量出现、outbox 堆积不可恢复 | 需石磊确认 |
| R-5 | 安全事件 | 越权访问可复现、service-jwt 绕过 | 武安确认后即时回滚 |
| R-6 | 人工裁决 | 白帆 / CTO / 人类决策者下达回滚指令 | 优先级最高 |

> 灰度阶段若仅金丝雀实例异常且流量可隔离，优先**摘除金丝雀**；全量或摘除非金丝雀后仍不达标，执行完整回滚（[`回滚预案.md`](回滚预案.md) §4）。

## 6. 阈值映射表（H/R ↔ P0/P1）

本表为灰度策略、回滚预案、监控基线三文档的**权威对齐索引**。执行时以**先触达**者为准。

### 6.1 暂停阈值映射（H-* ↔ P1）

| 灰度 H-* | 回滚预案阈值 | 监控 P1 | 对齐说明 |
|---|---|---|---|
| H-1 | 5xx > 1% 或较基线 +0.5% | P1-01（1%–5%） | H-1 为 P1-01 **下沿早警** |
| H-2 | gateway P95 > 800ms | P1-05（500ms–2s） | **800ms 衔接 P1-05 下沿，灰度暂停早警** |
| H-3 | `/readyz` 连续失败 | —（直达 P0-01） | readiness 失败快速升级 |
| H-4 | `*_consumer_lag_seconds` > 60s | P1-04（60s–300s） | **主指标秒级 lag**；条数 lag > 5000 仅辅助 |
| H-5 | 金丝雀探针失败 | P1-07（canary vs stable 2x） | 探针失败即时暂停 |

### 6.2 强制回滚映射（R-* ↔ P0）

| 灰度 R-* | 回滚预案阈值 | 监控 P0 | 对齐说明 |
|---|---|---|---|
| R-1 | 5xx > 5% | P0-02 | 一致 |
| R-2 | 建单成功率 < 99% | P0-03（< 95%） | **R-2 先于 P0-03 触发**（更严） |
| R-3 | 鉴权全面失效 | P0-04（SLI-02 > 20%） | 见下方换算 |
| R-4 | 数据一致性事故 | — | 人工 + 石磊确认 |
| R-5 | 安全事件 | — | 武安确认 |
| R-6 | 人工裁决 | — | 最高优先级 |

#### R-3 ↔ SLI-02 / P0-04 换算关系

监控基线 SLI-02 以**鉴权失败率**为主指标；回滚预案 R-3 以 **login 成功率**表述。换算如下：

| 表述 | 阈值 | 等价条件 |
|---|---|---|
| R-3（回滚预案） | `POST /api/v1/auth/login` 成功率 **< 95%** | login 路径鉴权失败率 **> 5%** |
| P0-04（监控基线） | SLI-02 鉴权失败率 **> 20%** | 全量鉴权尝试（含 refresh 等） |
| 灰度执行口径 | **统一以 SLI-02 为主判据** | SLI-02 > 5%（持续 2 min）→ 等同 R-3；SLI-02 > 20% → P0-04 |

> login 成功率 = `gateway_auth_attempts_total{result="success"} / gateway_auth_attempts_total`（login 路由子集）。执行时 PromQL 优先查 SLI-02 全量，login 子集作交叉验证。

### 6.3 业务 SLI 与告警分级总览

| 灰度门禁 | 对应 P1 | 对应 P0 | 说明 |
|---|---|---|---|
| SLI-01 ≥ 99.9% | P1-02（< 99.9%） | P0-03（< 95%） | 灰度门禁严于 P1 |
| SLI-03 < 60s | P1-04（60–300s） | P0-05（> 300s） | 秒级 lag 为主 |
| SLI-05 P95 < 500ms | P1-05（500ms–2s） | P0-06（> 2s） | PRD NFR 一致 |
| SLI-02 无突增 | P1-03（5%–20%） | P0-04（> 20%） | 见 §6.2 换算 |

## 7. 人类确认点（G1–G5）

按 **CTO 授权**，灰度放量的人类确认角色统一为 **CTO（Allen Li）**，与 [`发布监控告警基线.md`](发布监控告警基线.md) §7.1 C3 同句同角色。

| 门禁 | 时机 | 确认人 | 动作 |
|---|---|---|---|
| **G1** | 灰度启动前 | CTO | 确认 LKG 清单、本策略 v1.0 已签字、staging dry-run 计划 |
| **G2** | C1（5%）启动 | CTO | 确认预发验证通过、四服务镜像就绪 |
| **G3** | C1 → C2 推进 | CTO | 确认 C1 观测窗口 P0/P1 清零、SLI 达标 |
| **G4** | C2 → C3 推进 | CTO | 确认 C2 观测窗口 P0/P1 清零、SLI 达标 |
| **G5** | C3 全量（100%） | CTO | 确认 C3 观测 60 min 达标后提请全量；通过 `multica-test@hotmail.com` |

**铁律**：告警清零 ≠ 自动放量。万全提交观测报告后，由白帆提请 CTO 确认，CTO 回复后方可推进下一批。
