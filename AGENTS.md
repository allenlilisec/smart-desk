# SmartDesk 项目方法论补丁

> 版本：v1.0　|　日期：2026-06-14　|　编制：韩衡（质量管理团队）　|　裁决：CTO　|　状态：待文澜资料审视

本文件为 SmartDesk 项目的研发方法论补丁，基于 SUP-68（审核瓶颈防复发机制设计）的 CTO 裁决，固化以下方法论规则。

---

## 1. 交接闭环军规

**核心原则**：交接必须到人，且「接收人已认领并启动」才算交接到位，否则交接人不得收口。

### 1.1 交接门禁

1. 任一环节产出后，交接人必须在对应 issue 上 @ 指定接收人，写清下一步动作 + 验收标准。
2. 接收人「认领并启动」（issue 转 `in_progress` 或留开工评论）之前，交接人**不得收口/离开**。
3. PMO（程远/方通）每 15 分钟巡检每一条交接边，凡停在「已交接·未启动」的，立即催办；超 30 分钟升级 PMO Leader，超 60 分钟升级 CTO。

### 1.2 交接检查要点

| 检查项 | 标准 | 未达标动作 |
|---|---|---|
| 接收人明确 | 交接评论中已 @ 具体接收人 | 催办交接人补全 |
| 下一步动作清晰 | 写明具体动作 + 验收标准 | 催办交接人补全 |
| 接收人已启动 | issue 状态已变或开工评论已留 | 催办接收人；超 30min 升级 |

## 2. 并行评审机制

**核心原则**：评审权可分散并行，冻结签字权必须保持单一问责人。

### 2.1 资料评审并行

资料评审采用**文澜、沈思双线并行**，破沈思单点阻塞。但**评审人 ≠ 该模块终审签字人**（避免自评自签）。

**互审分工（默认）**：

| 模块 | 初审人 | 终审签字人 |
|---|---|---|
| core | 沈思 | 文澜 |
| insight | 文澜 | 沈思 |
| web | 沈思 | 文澜 |
| gateway | 文澜（已到最后一关）| 文澜 |

PMO（程远）可按负载微调分工，但终审签字人必须保持单一问责。

### 2.2 架构评审并行

架构评审由**梁栋主审**，秦诺并行做契约一致性核查（api-contract-check）加速，但秦诺的核查**不替代梁栋的设计裁决**。

### 2.3 并行推进规则

- 任一并行评审人通过即推进
- 异议走评论标注，不阻塞流程
- 评审人异议需明确标注问题点，由对应责任人修复后重审

---

## 3. 签字权问责机制

**核心原则**：「人人都能签」等于「没人负责」——签字权必须保持单一问责人。

### 3.1 签字权归属

| 签字类型 | 签字人 | 说明 |
|---|---|---|
| 资料终审签字 | 该模块终审责任人单人（文澜或沈思，按 §2.1 互审分工）| 单一问责，不可分散 |
| 设计/契约冻结签字 | 梁栋 | 架构唯一裁决人，秦诺并行核查不改变问责单点 |

### 3.2 签字人不可用时的处理

若某模块两人（文澜/沈思）都不可用导致签字单点失效：
1. 按 SLA 升级（见 §4）
2. 由 CTO 临时指派或代签

**禁止引入虚构责任人**：不得在方案中引入不在编的智能体（如「严兰」等）作为签字人。

## 4. 升级链路与 SLA

### 4.1 升级链路

```
审核人 → PMO（程远） → CTO → CEO（Allen）
```

**角色更正说明**：
- 何明是**质量管理 Leader**，不是 CTO
- 质量团队（韩衡）只做巡检度量与按链路升级，**不做设计/代码裁决**
- CTO 为单独的智能体角色，负责技术裁决与架构决策

### 4.2 升级 SLA

| 阶段 | 等待时长 | 升级动作 |
|---|---|---|
| T+15min | PR/评审任务无响应 | 自动催促（评论提醒审核人）|
| T+30min | 无进展 | 自动升级 PMO（程远）协调 |
| T+60min | 仍阻塞 | 自动升级 CTO 裁决 |
| T+90min | 极端阻塞 | CEO（Allen）介入 |

### 4.3 SLA 执行主体

- **巡检主体**：质量团队（韩衡）每 15 分钟执行停滞巡检
- **升级路径固化**：每次巡检发现阻塞即按表执行 @mention 升级
- **度量记录**：红黑台账记录每次升级事件及响应时长

## 5. 审核进度可视化机制

### 5.1 可视化项

| 可视化项 | 数据源 | 输出位置 |
|---|---|---|
| PR 等待时长排行榜 | GitHub PR API | 项目跟踪 issue 评论（每巡检周期）|

| 审核人负载分布 | Multica assignee 统计 | PMO 协调依据 |
| 阻塞链路图 | issue parent/dependency | 里程碑门禁预警 |

### 5.2 阈值预警

| 指标 | 阈值 | 预警动作 |
|---|---|---|
| in_review 占比 | >20% | 自动触发升级检查 |
| PR 等待时长 | >30min | 自动催促 + 升级 PMO |

## 6. 里程碑门禁预警机制

### 6.1 门禁监控

| 里程碑 | 门禁项 | 阻塞阈值 | 预警动作 |
|---|---|---|---|
| P2 冻结 | 4 模块详设评审通过 | 任一模块 >30min 无审核进展 | 升级架构团队 + PMO |
| P2 冻结 | 资料终审签字 | 终审签字人 >15min 无响应 | 升级资料团队 Leader |
| P3 启动 | P2 冻结确认 | P2 冻结后 >10min 未提升 P3 子任务 | 升级 PMO 派活 |

### 6.2 预警执行

- **门禁状态监控**：韩衡在流程合规核查中检查门禁状态
- **预警评论**：发现门禁阻塞立即在项目跟踪 issue 发布预警评论（标注阻塞链路）
- **自动提醒**：@mention 对应审核人 + 升级链路责任人

集体检视门禁清单主文档见 `quality/review-checklist.md`，由质量管理团队维护，CTO 裁决生效。本节保留索引与关联事件说明。

### 9.1 跨服务调用路径与 OpenAPI 契约一致

- **状态**：已生效（CTO 裁决，2026-06-18）
- **主文档**：`quality/review-checklist.md` 必检项 G-X
- **触发范围**：任何改动或新增跨服务调用的 PR 进入 `in_review` 时强制核对。
- **核心要求**：
  1. 调用方代码中的外部端点路径，与 `src/openapi/*.yaml` 中提供方契约逐字符一致；

## 10. Runtime 迁移治理规范

Runtime 迁移治理规范主文档见 `quality/runtime-migration-governance.md`，由质量管理团队（韩衡）维护，何明定稿、CTO 裁决生效。本节保留索引与关联事件说明。

### 10.1 规范范围

- 冲刺/demo 窗口冻结 runtime 迁移；
- 迁移完成后 5 分钟内派活检查；
- 迁移前 checklist；
- 红黑事件台账治理。

### 10.2 核心要求

1. 任何批量（3 个及以上）或影响关键路径的 agent runtime 迁移，必须避开 demo/冲刺/发布冻结窗口；
2. 紧急迁移须经 PMO Leader → CTO → CEO 书面批准，并执行分批、热备、值守、回滚、通知五类降级措施；
3. 迁移完成后 5 分钟内，PMO 必须完成派活检查，未通过按升级链路上报；
4. 迁移前必须完成 8 项 checklist 并记录在对应 issue 评论中；
5. 所有迁移相关事件须由韩衡在 30 分钟内录入红黑事件台账。

### 10.3 关联事件

- 本规范由 [SUP-347](mention://issue/1302fd35-9a4e-4d33-8ced-f970b4372765) demo 日 runtime 双切换致编码产能空窗黑事件（RB-20260619-001）触发，旨在防止迁移后产能空窗复发。
